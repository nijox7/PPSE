Lab 4 - Optimization of the simulation chain

Conditions d’achèvement

Task 0. Simulation throughput. Add a new metric to measure the throughput of the simulation in megabits per second (Mbps). For this, you simply need to multiply the the number of simulated frames (f_n) by the number of information bit K. Then you should divide the result by the time to complete one SNR point. In other words, Sim_thr = (f_n x K) / SNR_duration. The simulation throughput will become our main metric to evaluate the speed of our simulator.

Task 1. Per-block performance statistics. Measure the performance of each block (source, encoder, modulator, channel, demodulator and monitor). For each SNR and for each block you will compute:

  • The average, minimum and maximum latency of the block,
  • The average throughput of the block in Mbps. To do this you will always consider the number of output bits or symbols of the current block. Exceptionally for the monitor, you will consider the number of input bits K.
  • The average percentage of time in the simulation for the current block.

You will add a compiler definition ENABLE_STATS (that can be enabled from the compiler command line by doing -DENABLE_STATS), and these additional measures will be turned on only if this macro is defined.

Task 2. All zeros source. In order to reduce the latency to simulate one frame, implement the following simplified source:

// write only zeros in U_K
void source_generate_all_zeros(uint8_t *U_K, size_t K);

You will add the following command line parameter to enable it: --src-all-zeros. Of course, if --src-all-zeros is not set, you will run the initial source generator. Run the previously defined simulations (sim1 to sim5) with the --src-all-zeros enabled. Is there any difference in term of decoding performance? Is simulation throughput faster?

Task 3. All ones modulation. Still in order to reduce the latency to simulate one frame, implement the following simplified BPSK modulator:

// write only ones in X_N
void modem_BPSK_modulate_all_ones(const uint8_t *C_N, int32_t *X_N, size_t N);

You will add the following command line parameter to enable it: --mod-all-ones. When --mod-all-ones is set, skip the source execution. Run the previously defined simulations (sim1 to sim5) with the --mod-all-ones enabled. Is there any difference in term of decoding performance? Is simulation throughput faster?

Task 4. Fixed-point representation. In this task, you will implement the following quantizer module between the demodulator and the decoder:

// transform numbers from floating-point representation to fixed-point representation
// `s` is the number of bits used in the quantizer block
// `f` is the number of bits of the fractional part (`s` >= `f`)
void quantizer_transform8(const float *L_N, int8_t *L8_N, size_t N, size_t s, size_t f);
The quantizer will be enabled only if the the following parameter is set: --qf [f uint], otherwise, the simulation chain will stay the same as before (e.g. without the quantizer). --qf [f uint] range is [0;8]. When --qf [f uint] is set, and optional parameter can combine with it: --qs [s uint]. --qs [s uint] default value is 8 and its range is [1;8].

One may note that the previous repetition decoders are incompatible with L8_N datatype (int8_t). Thus, two new versions of the repetition decoder need to be implemented:
// hard decoder: first hard decides each LLR and then makes a majority vote
void codec_repetition_hard_decode8(const int8_t *L8_N, uint8_t *V_K, size_t K, size_t n_reps);
// soft decoder: computes the mean of each LLR to hard decide the bits
void codec_repetition_soft_decode8(const int8_t *L8_N, uint8_t *V_K, size_t K, size_t n_reps);
These two new versions will be selected with the following command line arguments:
  • -D "rep-hard8": for the hard decoder 8-bit,
  • -D "rep-soft8": for the soft decoder 8-bit.
Task 5. Decoding performance in fixed-point. Find {s,f} pairs that maximize the decoding performance and minimize s value (the number of bits after the quantizer). Find them for the previous simulations (e.g. from sim1 to sim5). For each simulation, give the best {s,f} pair. Once you found them, plot the 3 same graphs as in Lab 3 where you will add the quantized decoder (8 plots per graphs).

Task 6. Speedup the repetition decoders with SIMD. Implement the following decoder based on NEON SIMD instructions saw in class:
// hard decoder: first hard decides each LLR and then makes a majority vote
void codec_repetition_hard_decode8_neon(const int8_t *L8_N, uint8_t *V_K, size_t K, size_t n_reps);
// soft decoder: computes the mean of each LLR to hard decide the bits
void codec_repetition_soft_decode8_neon(const int8_t *L8_N, uint8_t *V_K, size_t K, size_t n_reps);
We will always consider that K is a multiple of 16. The vectorization strategy is to process 16 elements per 16 elements instead of one by one. Before to call NEON intrinsic functions you need to include arm_neon.h header that contains the function declarations. Then, be sure to compile with -03 and -march=native optimization flags. Here is the sub-set of intrinsic functions you can use (and it is forbidden to use other NEON intrinsic functions):
  • int8x16_t vdupq_n_s8(int8_t value)
  • int8x16_t vsetq_lane_s8(int8_t a, int8x16_t v, const int lane)
  • int8_t vgetq_lane_s8(int8x16_t v, const int lane)
  • int8x16_t vld1q_s8(int8_t const * ptr)
  • void vst1q_s8(int8_t * ptr, int8x16_t val)
  • int8x16_t vaddq_s8(int8x16_t a, int8x16_t b)
  • int8x16_t vqaddq_s8(int8x16_t a, int8x16_t b)
  • uint8x16_t vcltzq_s8(int8x16_t a)
  • uint8x16_t vceqq_s8(int8x16_t a, int8x16_t b)
  • int8x16_t vandq_s8(int8x16_t a, int8x16_t b)
These two new versions will be selected with the following command line arguments:
  • -D "rep-hard8-neon": for the hard decoder 8-bit in SIMD NEON,
  • -D "rep-soft8-neon": for the soft decoder 8-bit in SIMD NEON.
To validate your implementation you will run the following simulations for the hard repetition decoder (Eb/N0 = 14 dB -> FER @ 5.0e-06):
  • ./simulator -m 0 -M 14 -s 1 -e 100 -K 32 -N 8192 -D "rep-hard"
  • ./simulator -m 0 -M 14 -s 1 -e 100 -K 32 -N 8192 -D "rep-hard8"      --qf [f] --qs [s]
  • ./simulator -m 0 -M 14 -s 1 -e 100 -K 32 -N 8192 -D "rep-hard8-neon" --qf [f] --qs [s]

Be careful, for the three previous simulations, do not use the --mod-all-ones parameter because of the bias in the hard decoder when the number of votes is even!

And for the soft decoder (Eb/N0 = 11 dB -> FER @ 9.5e-06):

  • ./simulator -m 0 -M 11 -s 1 -e 100 --mod-all-ones -K 32 -N 8192 -D "rep-soft"
  • ./simulator -m 0 -M 11 -s 1 -e 100 --mod-all-ones -K 32 -N 8192 -D "rep-soft8"      --qf [f] --qs [s]
  • ./simulator -m 0 -M 11 -s 1 -e 100 --mod-all-ones -K 32 -N 8192 -D "rep-soft8-neon" --qf [f] --qs [s]
Groups of 3 simulations should have close FER. Comment the achieved decoder throughput.
